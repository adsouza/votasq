# Migrate Flutter client to Swift Package Manager (hybrid)

**Date:** 2026-05-01
**Scope:** `apps/client/ios/`, `apps/client/macos/`, root `pubspec.yaml`, `.claude/CLAUDE.md`
**Status:** Design approved; implementation plan to follow.

## Context

The Flutter client (`apps/client`) targets iOS, Android, web, macOS, and Windows. The Apple platforms (iOS, macOS) currently use CocoaPods to manage native dependencies for Flutter plugins (Firebase, Google Sign-In, ML Kit, shared_preferences). Flutter 3.41 ships stable Swift Package Manager (SwiftPM) support for these platforms, in opt-in hybrid mode where SwiftPM and CocoaPods coexist: plugins migrate to SwiftPM where they support it, while plugins without a `Package.swift` continue to be installed via the Podfile.

Migrating reduces dependency-resolution time, brings the project in line with Flutter's preferred default for new projects (3.32+), and eliminates a category of CocoaPods-specific issues (slow `pod install`, Ruby gem version drift).

## Goals

- Enable Flutter SwiftPM support for both iOS and macOS in hybrid mode.
- Preserve current behavior: same deployment targets, same plugin set, same app IDs and entitlements, same flavors (development/staging/production).
- Drop the redundant macOS `post_install` deployment-target override; keep the rest of the Podfile structure.
- Verify the migration with build, Xcode test targets, `flutter test`, and a smoke launch on both platforms.
- Document the per-machine config requirement in `.claude/CLAUDE.md` and automate it in `melos setup`.

## Non-goals

- Not migrating Android, web, Linux, or Windows builds.
- Not bumping deployment targets, the Flutter version, or our `pubspec.yaml` plugin version constraints. Note that native iOS/macOS SDK versions (e.g., the underlying `firebase-ios-sdk`) are determined by the dependency resolver, not by us â€” switching from CocoaPods to SwiftPM re-resolves these against the SwiftPM `Package.swift` ranges declared by each Flutter plugin, which can shift the resolved version within the constraint. Observed: `firebase-ios-sdk` resolved at 12.9.0 under CocoaPods, 12.12.1 under SwiftPM (same major, three minor versions newer). This is expected behavior of the resolver swap, not a deliberate version bump.
- Not touching CI. Current CI does not build iOS or macOS, so SwiftPM enablement has no CI impact in this PR.
- Not removing CocoaPods entirely. Hybrid mode is the chosen end state; the Podfile stays.
- Not migrating the server (`apps/server`) or `packages/shared`.

## Key decisions

| Decision            | Choice                                                | Rationale                                                                                                                                                                                                                                |
|---------------------|-------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Migration mode      | Hybrid (SwiftPM + CocoaPods coexist)                  | Lower risk; Flutter's tooling routes plugins automatically; CocoaPods remains as fallback for plugins lacking `Package.swift`.                                                                                                           |
| Deployment targets  | Keep current values: iOS 15.5, macOS 10.15            | These floors match the SDKs' actual requirements today (verified by inspection of `Pods/Pods.xcodeproj/project.pbxproj`). The current Podfile request is already at the SDK-driven floor.                                                |
| Sequencing          | Single-shot dual-platform migration                   | Both platforms in one PR. The Flutter SwiftPM tooling is mature and reversible. Two-phase migration would duplicate the same diff with marginal benefit.                                                                                 |
| Verification bar    | Build + automated tests + smoke launch                | Catches missing-resource and Info.plist-privacy regressions that builds alone do not surface. RunnerTests are no-op XCTests today, but compiling and running them validates the test target's framework/module resolution under SwiftPM. |
| Collaborator opt-in | Document in `CLAUDE.md` and automate in `melos setup` | `flutter config` is per-machine. Documenting it gives future contributors (and Claude in future sessions) the answer; `melos setup` automation makes it harder to forget.                                                                |

## Verification plan

Steps run in order. Stop and triage on the first failure.

1. **Pre-flight baseline.** With SwiftPM disabled (current state), run `flutter clean && flutter build ios --debug --flavor development --no-codesign --target lib/main_development.dart` and the macOS equivalent. Run `flutter test` in `apps/client/`. Capture results. If the baseline is broken, fix it before migration begins.
2. **Enable SwiftPM.** `flutter config --enable-swift-package-manager`.
3. **Clean rebuild iOS.**
   - `flutter clean` in `apps/client/`.
   - `cd apps/client/ios && pod deintegrate && rm -rf Pods Podfile.lock`.
   - `flutter build ios --debug --flavor development --no-codesign --target lib/main_development.dart`.
   - `flutter build ios --release --flavor production --no-codesign --target lib/main_production.dart`.
   - We test development+debug and production+release; the staging flavor is skipped to keep the matrix small. If staging diverges in build settings (it currently does not), add it.
   - Inspect post-build `Podfile.lock` to confirm pod count dropped and the expected plugins migrated to SwiftPM.
4. **Clean rebuild macOS.** Same pattern as iOS, with `flutter build macos --debug --flavor development --target lib/main_development.dart` and `flutter build macos --release --flavor production --target lib/main_production.dart`.
5. **Run Xcode test targets** on both platforms.
   - iOS: `xcodebuild test -workspace apps/client/ios/Runner.xcworkspace -scheme Runner -destination 'platform=iOS Simulator,name=iPhone 15'`.
   - macOS: `xcodebuild test -workspace apps/client/macos/Runner.xcworkspace -scheme Runner -destination 'platform=macOS'`.
   - These compile and run `RunnerTests` (currently a no-op XCTest, but verifies the test target's framework/module resolution under SwiftPM).
6. **Run `flutter test`** in `apps/client/` to confirm Dart-side tests still pass.
7. **Smoke launch.**
   - iOS simulator: `flutter run --flavor development --target lib/main_development.dart`.
   - macOS: `flutter run --flavor development --target lib/main_development.dart -d macos`.
   - Confirm: app launches without a native crash, Firebase initializes (no crash on `Firebase.initializeApp`), the initial screen renders. Catches missing resources, missing Info.plist privacy entries, framework load failures.
8. **Format and analyze.**
   - `dart format --set-exit-if-changed apps packages`.
   - `flutter analyze apps packages`.

### Definition of done

- Steps 3-8 all pass.
- Diff to both `project.pbxproj` files reviewed; only SwiftPM-related changes present (no unintended `CODE_SIGN_*` or entitlements-file edits).
- `Podfile.lock` files committed in their post-migration form.
- macOS `post_install` deployment-target override removed.
- `pubspec.yaml` `melos setup` script updated.
- `.claude/CLAUDE.md` updated.

## Risks and mitigations

1. **A plugin lacks `Package.swift`** (most likely candidates: `google_mlkit_*`). Hybrid mode handles this transparently; affected plugins remain in `Podfile.lock`. Detection: post-migration `Podfile.lock` still lists the plugin. Not a blocker.
2. **macOS code signing or entitlements drift.** Xcode rewriting `project.pbxproj` can shift `CODE_SIGN_*` settings or `*.entitlements` references. Mitigation: review the `project.pbxproj` diff carefully before commit, focusing on signing- and entitlements-related lines. Reject any unintended changes.
3. **Stale `~/Library/Developer/Xcode/DerivedData`** can mask real failures. Mitigation: clear DerivedData for the Runner project before each clean rebuild step.

## Rollback

If verification fails and the issue cannot be quickly resolved:

```sh
flutter config --no-enable-swift-package-manager
git checkout -- apps/client/ios apps/client/macos pubspec.yaml .claude/CLAUDE.md
cd apps/client/ios && pod install
cd ../macos && pod install
```

The repo returns to the pre-migration state. SwiftPM enablement is a per-machine flag with no committed-state coupling, so the rollback is local-only.

## Open questions

None at design time. Plugin-specific SwiftPM gaps (if any) are discovered during step 3-4 of the verification plan and handled by hybrid mode automatically; no design change required.
